# ADR-0001: 桌面端 + 浏览器插件混合架构

## 状态

已接受

## 背景

零职达需要同时满足以下需求：
- 本地安全存储用户敏感信息（简历数据）
- AI简历生成（PDF/DOCX输出）
- 与招聘网站交互（读取JD、跳转投递）
- 定时后台任务（岗位爬取）

单一技术形态（纯Web、纯桌面、纯插件）均无法同时满足以上所有需求。

## 决策

采用**桌面端 (Tauri) + 浏览器插件 (Manifest V3)** 混合架构。

### 职责划分

| 功能 | 桌面端 | 浏览器插件 |
|------|--------|-----------|
| 用户数据存储 | SQLite本地数据库 | 无 |
| AI模型调用 | Ollama本地 + 第三方API | 无 |
| 简历生成 (PDF/DOCX) | Rust后端生成 | 无 |
| 岗位定时爬取 | Playwright无头浏览器 | 无 |
| 证件照存储 | 本地文件系统 | 无 |
| JD读取 | 无 | 内容脚本注入招聘页面 |
| 跳转投递 | 无 | 新标签页打开投递页 |
| 通信 | 启动localhost HTTP服务 | fetch请求桌面端服务 |

### 通信机制

桌面端启动本地HTTP服务器（默认端口 `localhost:8765`），插件通过标准HTTP请求与之通信：

```
插件 ──HTTP──► 桌面端localhost:8765
  │              │
  │              ├─► SQLite读写
  │              ├─► AI模型调用
  │              ├─► 文件系统操作
  │              └─► 简历生成
```

### 安全设计

- **绑定localhost**：HTTP服务只绑定 `127.0.0.1`，不对外暴露
- **端口冲突处理**：默认端口8765，若被占用则自动尝试8766-8770
- **可选Token验证**：启动时生成随机Token，插件请求需携带Token头，防止其他恶意应用访问

## 备选方案

### 方案A：纯浏览器插件
- **拒绝原因**：无法本地运行AI模型（必须依赖远程API，用户成本高）；无法定时后台爬取；本地存储受限（IndexedDB容量和查询能力不如SQLite）

### 方案B：纯桌面端（内置浏览器）
- **拒绝原因**：Tauri内置WebView无法完美模拟Chrome扩展的内容脚本注入机制；招聘网站反爬策略可能识别内置浏览器指纹

### 方案C：Electron替代Tauri
- **拒绝原因**：Electron包体积大（>100MB），Tauri包体积小（<10MB）；Tauri基于Rust，PDF生成等后端任务性能更好

## 后果

### 正面
- 功能完整：各端只做擅长的事
- 用户体验好：数据本地存储，无需登录
- 性能优秀：Tauri轻量，Rust后端高效
- 安全可靠：本地服务不对外暴露，敏感操作受控

### 负面
- 开发复杂度高：需维护两个代码库
- 通信延迟：插件与桌面端通过HTTP通信，存在毫秒级延迟
- 安装门槛：用户需同时安装桌面端和浏览器插件
